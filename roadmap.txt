1. Transaction Processing & Reporting System 

Concept: 
A system to record, track, and report financial transactions (e.g., sales, purchases, payments) for a business. Users can add, update, delete, and query transactions. Reports summarize daily, weekly, or monthly activity for decision-making.


How it covers everything:
The project uses PostgreSQL to store transaction data, including details like amounts, dates, categories, and descriptions, as well as optional tables for users and transaction categories. Using FastAPI and data modeling, transactions, categories, and optionally users are defined as models with Pydantic schemas to validate incoming data and shape responses. PostgreSQL and SQL  serves to perform all database operations, such as creating, reading, updating, and deleting transactions, and to query aggregated data for reporting. The system exposes multiple RESTful API endpoints to handle transactions, categories, and report generation, following proper HTTP methods and response codes. As a full stack exercise, the FastAPI backend is fully functional, and endpoints can be tested through tools like Postman or a React.JS/HTML/CSS frontend. For an extra challenge, the system can implement JWT-based authentication to manage user access, generate downloadable reports, and provide summary endpoints for analytics.


Example scenario: Real-World Inspiration of how the system gonna work
Let’s assume your project is inspired by a computer shop retail business (like PC Express, UNI PC, or DataBlitz):
  --  Your ordering system: PC EDRIAN EXPRESS → handles product catalog, sales orders, and payments.

  -- Your transaction system (this project): EDRI: Transaction Processing & Reporting System → tracks all money flow from the business, independent of the ordering system.


Transactions categories for the business:

1. Sales: Total revenue from selling PCs, components, and accessories.
    -- Can report weekly, daily, or monthly — bulk reporting is fine for this system.

2. Purchases: Money spent buying items from suppliers (NVIDIA, AMD, other PC parts suppliers).

3. Expenses: Operational costs not tied to purchases, e.g., rent, utilities, furniture, on-site store maintenance.

4. Salary: Payments to employees (baristas, cashiers, on-site staff).
    -- No need to implement a full payroll system; just record payments.


Workflow with users and admin:

-- Admin: Full control — can CRUD any transaction, manage categories, generate reports, manage users, and approve/decline new admin requests.

-- Standard users (managers): Can CRUD their own transactions and view their own summaries.

-- Admin registration: For the demo, only you are admin initially. If someone registers as admin, you receive a notification to approve or reject. Standard users can register freely.

-- Roles are flexible for the demo: Users may select any category when adding transactions, but you can show they are “assigned” basic responsibilities (e.g., User 1 → purchases, User 2 → expenses).



Database:PostgreSQL 
- Store all transaction data: amount, date, type (credit/debit), category, description.
- Store related tables: users (if authentication is implemented), transaction categories, reports history.
- Can include relational constraints: e.g., each transaction belongs to a category, optional user table.


FastAPI & Data Modeling: 
- Models for transactions, categories, and optionally users.
- Define Pydantic schemas for request validation and response formatting.
- Include relationships between models (e.g., transactions → categories).


SQL scripts with PostgreSQL: 
- layer to interact with PostgreSQL.
- CRUD operations: Create, Read, Update, Delete transactions.
- Query filtering for reports (e.g., sum totals per category or per time period).


API / REST: 
Endpoints for:
/transactions → list, create, update, delete transactions.
/categories → manage categories.
/reports → generate reports (daily, monthly, etc.).
- Use REST principles: proper HTTP methods (GET, POST, PUT, DELETE).


Full Stack: FastAPI serves as backend.
- Configured environment includes: FastAPI app, PostgreSQL, SQL, Pydantic models, API routes.
- clean env.file, clean folder structures etc.
